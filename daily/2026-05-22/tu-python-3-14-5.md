# Python 3.14.5 — 増分 GC のロールバックと「本番で詰まった」教訓

- **日付**: 2026-05-22
- **カテゴリ**: Language
- **ソース**: [Python Insider (RC1)](https://blog.python.org/2026/05/python-3145rc1/), [pydevtools 解説](https://pydevtools.com/blog/python-3145-rolls-back-the-incremental-garbage-collector/), [python.org Discuss](https://discuss.python.org/t/reverting-the-incremental-gc-in-python-3-14-and-3-15/107014)

## 概要

Python 3.14.5 が 2026-05-10 にリリースされた。マイナーパッチ番号にもかかわらず、**3.14 系で導入された増分ガベージコレクタ (incremental GC) を 3.13 と同じ世代別 GC へ巻き戻す** という重い変更が入っている。パッチリリースで実装の主要モジュールを差し戻すのは極めて異例で、本番環境からの「メモリが膨らんでいる」というレポートが押し戻した形だ。

## 主な変更点

### 増分 GC の巻き戻し

3.14.0 で Mark Shannon の incremental collector がマージされ、旧世代を細切れに走査することで GC ポーズを抑える仕組み (世代数は 3 → 2) になっていた。リリースノートでは大きなヒープで「ポーズ時間が桁オーダーで短くなる」とアピールしていた。

しかし 3.14.0〜3.14.4 を本番投入したチームからの報告では:

- ポーズは確かに短くなった
- が、**メモリ内のゴミの総量は減らず、オブジェクトが旧来より長く生き延びる**
- 結果として常駐メモリが最大 5 倍程度に膨らむケースが報告された
- 循環参照を多用するワークロード (ORM, Django など) で特に顕著

Adam Johnson 氏は Django のメモリ「リーク」を 3.14 の incremental GC のせいだと特定する記事を 2026-04 に公開している ([リンク](https://adamj.eu/tech/2026/04/20/django-python-3.14-incremental-gc/))。

### 3.14.5 と 3.15 でも巻き戻し

- 3.14.5: 増分 GC を取り除き、3.13 と同じ世代別 GC (3 世代) を復帰
- 3.15 (開発中) でも同様に巻き戻すことが [Discuss スレッド](https://discuss.python.org/t/reverting-the-incremental-gc-in-python-3-14-and-3-15/107014) で合意された
- その他、約 113 件のバグ修正、ビルド改善、ドキュメント更新

### 影響

- アップグレードしたメリットを「ポーズ削減」目的で享受していたユーザは特性が変わるので注意
- メモリ膨張に悩んでいたチームには明確な恩恵があり、3.14.5 への追従が強く推奨される
- C 拡張側で GC API (世代数や閾値) に依存していた場合の挙動も 3.13 互換に戻る

## 破壊的変更・移行ガイド

- API レベルで「世代数」を直接参照するコード (例: `gc.get_count()` の返却値の長さなど) は 3.14.0〜3.14.4 の挙動から戻る
- アプリの再ビルドや依存更新は不要、`pip install --upgrade` で取り込める
- 性能/メモリ特性が変わるため、ベンチマークを取り直すこと
- 3.14 系で「ポーズが短い」前提の SLO を組んでいた場合は SLO の見直しが必要

## 今後の注目点

- 「**増分 GC をパッチでロールバック**」というのは CPython の歴史でも稀。今後 GC を入れ替えるプロセス (本番試験、フリーバージョン期間) を仕組み化する議論につながる可能性
- Mark Shannon らは別のアプローチで GC ポーズ問題に再挑戦する見込み。3.16 ではどう設計を変えるか
- Free-threaded / JIT など 3.14 で同時に supported になった他の機能は引き続き安定運用フェーズ。GC 議論と切り分けて検証する必要がある
- ORM 系フレームワークの設定で「GC 設定をユーザに開放するべきか」という議論も再燃しそう
