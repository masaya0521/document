# マネーフォワード GitHub 不正アクセス事件: 本番カード情報がソースコードに混入する開発体制の落とし穴

- **日付**: 2026-05-10
- **カテゴリ**: Japan Tech
- **ソース**:
  - [マネーフォワード公式 (第一報)](https://corp.moneyforward.com/news/info/20260501-mf-press-1/)
  - [マネーフォワード クラウドサポート (5/7 追記)](https://biz.moneyforward.com/support/news/20260501.html)
  - [Zenn — エンジニア視点の解説](https://zenn.dev/awesome_kou/articles/moneyforward-github-source-leak)
  - [Impress Watch](https://www.watch.impress.co.jp/docs/news/2106374.html)
  - [セキュリティ対策 Lab](https://rocket-boys.co.jp/security-measures-lab/github-unauthorized-access-moneyforward/)

## 概要

2026 年 5 月 1 日、家計簿・会計クラウド大手のマネーフォワードが、自社で利用している GitHub への不正アクセスを公表した。攻撃者は漏えいした GitHub 認証情報を悪用してソースコードリポジトリをコピーし、その中に「本来そこにあるべきではなかった」本番カード保持者情報 370 件 (氏名アルファベット表記とカード番号下 4 桁) と各種シークレット (API キー、DB パスワード等) が含まれていたことが、後続調査で明らかになった。CVV やカード番号全桁、有効期限の流出は確認されていない。

事件は単なる「アカウント乗っ取り」では収まらず、**現代の SaaS 企業の多くが抱える「本番データがリポジトリ内に滲み出す」構造的リスク**を改めて突きつけた。マネーフォワードはマネーフォワード ME / クラウド会計 / クラウド給与 / マネーフォワードビジネスカード等を運営しており、銀行口座連携機能を一時停止して再発防止と影響調査を進めている。

### 何が流出したのか

公表ベースで「流出した可能性がある」とされたのは:

1. **マネーフォワード ビジネスカード 370 件**: カード保持者名 (アルファベット) とカード番号下 4 桁
2. **ソースコード**: マネーフォワードグループ各サービスのリポジトリ全体
3. **各種認証情報**: ソースコード内に埋め込まれていた API キー、DB 接続情報等

CVV や全桁のカード番号は決済処理側 (PCI DSS 準拠の別環境) で保持しており、ソースリポジトリ側には存在しなかったとされる点が、被害規模を抑える要因となった。

### なぜ本番データがリポジトリに混入したのか

マネーフォワード自身の説明は「個人情報の取り扱いを伴うサービスの更新作業を行う過程で、個人情報が含まれたファイルが本来の管理手順から外れて誤って GitHub 上に保管されていた」というもので、根本原因の特定には至っていない。Zenn の解説記事は、よくあるパターンとして以下を挙げている:

- **テストフィクスチャの誤用**: CI / E2E テスト用に本番データのスナップショットを流用
- **障害調査時の一時ダンプ**: バグ再現のために本番 DB から取得したダンプが消去されずコミットされる
- **マイグレーションスクリプト**: 環境変数化されるべき認証情報がスクリプトにベタ書き

特に金融系のサービスでは「本番でしか再現しない不具合」を再現するために、開発者が本番に近いデータを手元に持ち込みやすい構造があり、これが今回の事件の温床になったとみられる。

### Git 履歴という構造的リスク

仮に最新の `main` ブランチがクリーンに保たれていたとしても、攻撃者は `git log -p --all` で全ブランチ・全タグの過去差分を辿り、削除済みコミットからシークレットを復元できる。これが「ソースコードがコピーされた」=「履歴ごとコピーされた」ことを意味する重大さである。

シークレットを誤って commit してしまった場合、`git rm` ではなく `git filter-repo` や BFG Repo-Cleaner で**履歴自体を書き換える**必要があるが、共有リポジトリで履歴を強制書き換えるのは現実的に困難なケースが多い。よって「**コミットしてしまった瞬間に、そのシークレットは無効化と再発行が必須**」になる。

### 初動対応の評価

公表ベースで確認できる対応は:

- 不正アクセス経路となった認証情報の無効化、アカウント遮断 (即日完了)
- ソースコードに含まれる各種認証キー・パスワードの無効化と再発行 (5/7 時点でほぼ完了)
- 銀行口座連携機能の一時停止 (5/10 時点で継続中)
- カード保持者への個別通知 (5/3 以降順次)

**速報性 (発覚から 1 日以内) と影響範囲特定の段階的開示**は適切だった一方、「なぜ本番データが入り込んだのか」「なぜ事前に検知できなかったのか」の説明はまだ不十分で、第三者調査委員会の報告書を待つ必要がある。

## 業界への教訓

### 1. シークレットスキャンの一次防衛線化

GitHub Advanced Security の Push Protection、`gitleaks`、`TruffleHog` 等を **pre-commit hook + CI の両方** に組み込み、シークレットがコミット可能性ゼロにする運用が必須である。GitHub の secret scanning は無料枠でも公開リポジトリに対してパートナープログラム経由で動作するが、private リポジトリは Enterprise 契約が必要である点にも注意。

### 2. 本番データを開発者環境に持ち込まない

- **合成データ生成**: Faker, Factory Bot, mimesis 等で本番に近い分布のテストデータを生成
- **PII マスキング**: 本番 DB のスナップショットを開発で使う際は、コピー時点で必ずマスキング処理を挟む
- **ダウンロード可能環境の隔離**: PII を含むデータは隔離された分析環境のみで参照可能とし、ローカルへのダウンロードを禁止

### 3. シークレット管理の脱コミット化

- AWS Secrets Manager / HashiCorp Vault / Doppler 等への完全移行
- アプリケーション側は **起動時 / リクエスト時に取得** し、ファイル化しない
- CI/CD は OIDC ベースの短期トークン (GitHub Actions の `aws-actions/configure-aws-credentials` 等) で長期 IAM キーを撤廃

### 4. アクセス制御の最小化と細粒度化

- Personal Access Token (PAT) は fine-grained PAT に統一し、リポジトリ単位・権限単位で発行
- SSO/SAML 強制と、退職者の即時アカウント無効化フロー
- リポジトリへの管理者権限は最小化し、設定変更を Audit Log で監視

### 5. 履歴の前提を疑う

「最新コードはクリーン」だけでは不十分で、**履歴ごと公開されることを前提にした設計**が必要。シークレットは「コミットされた瞬間に侵害されたものと見なす」運用ルールが現実的である。

## 今後の注目点

- **第三者委員会報告書の内容**: 漏えいの根本原因、影響時期、横展開の有無
- **金融庁・個人情報保護委員会の対応**: 行政指導・処分の可能性
- **PCI DSS / SOC2 等の認証への影響**: 監査上の指摘事項として残るかどうか
- **銀行口座連携機能の再開時期**: 影響を受けたユーザは多く、長期停止は事業影響が大きい
- **業界全体への波及**: 同様の構造的リスクを抱える他社の対応 (シークレットスキャン強化等)
- **エンジニア採用市場への影響**: 「セキュリティ重視」を打ち出す企業との差別化が進むか
