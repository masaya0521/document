# GlassWorm マルウェア — GitHub サプライチェーン攻撃の全容

- **日付**: 2026-03-19
- **カテゴリ**: Tech
- **ソース**: [The Hacker News](https://thehackernews.com/2026/03/glassworm-attack-uses-stolen-github.html)

## 概要

GlassWorm は、盗まれた GitHub トークンを使って Python リポジトリにマルウェアを force-push で注入するサプライチェーン攻撃キャンペーン。2026年3月8日以降、GitHub、npm、VS Code/Cursor 拡張機能を含む433件以上のコンポーネントが侵害された。Django アプリ、ML 研究コード、Streamlit ダッシュボード、PyPI パッケージなど幅広い Python プロジェクトが標的となっている。

## 詳細

### 攻撃手法（ForceMemo）

攻撃は4段階のプロセスで実行される:

**第1段階: 初期侵害**
悪意のある VS Code および Cursor 拡張機能を通じて開発者のシステムを侵害し、GitHub トークンを窃取する。

**第2段階: リポジトリ改ざん**
盗んだ認証情報でリポジトリにアクセスし、`setup.py`、`main.py`、`app.py` などの Python ファイルに難読化されたマルウェアコードを挿入。正規のコミットをリベースした上で force-push を行い、元のコミットメッセージ・著者名・著者日時を維持することで改ざんを隠蔽する。

**第3段階: ペイロード配信**
- Base64 エンコードされたペイロードを展開
- ロシアローカライズのシステムを検出した場合はスキップ（ロシア関連の攻撃者であることを示唆）
- Solana ウォレット（`BjVeAjPrSKFiingBn4vZvghsGj9KCE8AJVtbc9S8o8SC`）のトランザクションメモフィールドから C2 サーバーの URL を取得するという独自の手法を使用

**第4段階: 追加ペイロードの実行**
暗号化された JavaScript をダウンロードし、暗号資産とデータの窃取を実行。npm ペイロードはディスクに書き込まず、`eval()` または Node.js の `vm.Script` サンドボックス内でメモリ上実行。48時間の再実行防止ロックを `~/init.json` に設定。

### 被害規模とタイムライン

| 日付 | イベント |
|------|---------|
| 2025-11-27 | C2 アドレスの最初のトランザクションを確認 |
| 2026-03-03〜09 | インビジブル Unicode 文字を使った151+リポジトリの侵害（第一波） |
| 2026-03-08 | GitHub Python リポジトリへの初期注入を確認 |
| 2026-03-16 | React Native npm パッケージの悪意あるバージョンを検出 |
| 2026-03-19 | 累計433件以上の侵害コンポーネントを確認 |

### 影響を受けたエコシステム

- **GitHub**: Python リポジトリ（Django、ML、Streamlit、PyPI パッケージ）
- **npm**: React Native パッケージ2件が直接侵害
- **VS Code / Cursor**: 悪意ある拡張機能がトークン窃取の入口
- **OpenVSX**: VS Code 拡張機能マーケットプレイスも影響

### 検知を困難にする技術

- Git 履歴の改ざんにより PR やコミット追跡を回避
- 元のコミット情報（メッセージ・著者・日時）を完全に保持
- インビジブル Unicode 文字によるペイロードの隠蔽（別波キャンペーン）
- Solana ブロックチェーンを C2 通信に利用する非従来型の手法

### 発見者

Aikido Security、Socket、Step Security、OpenSourceMalware コミュニティが共同で調査・報告。

## 今後の注目点

- GitHub のトークン管理ポリシーの強化（fine-grained PAT への移行推進など）
- force-push の検知・アラート機能の改善（GitHub の監査ログの拡充）
- IDE 拡張機能マーケットプレイスにおけるセキュリティ審査の厳格化
- ブロックチェーンを C2 に利用する手法の検知技術の発展
- Python / npm エコシステムにおけるサプライチェーン攻撃の防御策の進化（sigstore 署名の普及など）
- 開発者個人レベルでの対策: トークンの定期ローテーション、拡張機能の監査、リポジトリの force-push 制限設定
