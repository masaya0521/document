# Rust 1.95 — `cfg_select!` マクロで条件付きコンパイルを式に

- **日付**: 2026-04-16（リリース）
- **カテゴリ**: Language
- **ソース**: [Phoronix: Rust 1.95 Released](https://www.phoronix.com/news/Rust-1.95-Released), [std::cfg_select ドキュメント](https://doc.rust-lang.org/stable/std/macro.cfg_select.html), [Rust Changelogs](https://releases.rs/)

## 概要

Rust 1.95.0 が 2026-04-16 にリリースされた。最大の目玉は標準ライブラリに組み込まれた `cfg_select!` マクロの安定化で、`cfg-if` クレートが長らく担っていた「設定述語のコンパイル時マッチ」を依存ゼロかつ式コンテキストでも書ける形で実現した。これに加えて `if let` ガード周辺のスマート化、いくつかの新規 stable API、および埋め込み開発者向けの破壊的変更が含まれる。

## 主な変更点

### 1. `cfg_select!` マクロの安定化

複数の `cfg(...)` 述語をコンパイル時に評価し、最初に true となったブランチのコードを発生させるマクロ。`std`／`core` に式マクロとして定義されており、外部クレート不要で使える。

**アイテムレベルの例（`cfg-if` 互換の使い方）:**

```rust
cfg_select! {
    unix => { fn foo() { /* unix specific functionality */ } }
    target_pointer_width = "32" => { fn foo() { /* 32-bit fallback */ } }
    _ => { fn foo() { /* generic fallback */ } }
}
```

**式コンテキストでの利用:**

```rust
let label = cfg_select! {
    unix => "unix",
    windows => "windows",
    _ => "other",
};
```

ポイントは「式として書ける」こと。これまで `cfg-if` だとアイテム生成が前提で、関数の中で値だけを切り替えたい用途にはやや過剰だった。`cfg_select!` は文・式どちらでも使えるため、定数文字列やビット幅を切り替える小さな分岐にも自然にフィットする。

`_` がフォールバック分岐で、`match` のワイルドカードと同じ感覚。網羅性が静的に保証されるわけではないが、最初に true になった分岐だけがコード生成されるので、`#[cfg(...)]` を散りばめる従来の方式より見通しが良くなる。

### 2. `if let` ガードのスマート化

`match` アーム内の `if let` ガードや一時値の取り扱いが改善され、以前なら借用チェッカに弾かれていた書き方が通るケースが増えた。具体的にはガード内で生成した一時値の生存範囲が直感に近づき、`Option<&T>` を絡めたパターンマッチが書きやすくなっている。

### 3. 新規 stable API

各種コレクション・整数型・`std::sync` 周辺で複数の API が安定化。詳細は [Rust 1.95 リリース告知](https://blog.rust-lang.org/) と `releases.rs` のチェンジログを参照。

### 4. 破壊的変更（埋め込み開発向け）

1.86 で deprecated 化されていた `i586-pc-windows-msvc` ターゲットなど、いくつかの低 tier ターゲットに関するクリーンアップが入っている。古い 32-bit Windows 向けの組み込み／レガシーバイナリをビルドしている場合は、Tier 表とビルドマトリクスを確認したい。

## 移行ガイド

破壊的影響は限定的で、ほとんどのプロジェクトは `rustup update stable` だけで済む。ただし以下に該当する場合は確認推奨。

- **`cfg-if` クレート利用中**: そのままでも動くが、新規コードは `cfg_select!` への置き換えを検討。`cfg-if` 依存を削れる Cargo グラフのスリム化メリットがある
- **古い 32-bit Windows ターゲット**: 1.86 以降 deprecated だったターゲットの削除フェーズに入っているので、ビルドマトリクスから除外可能か再評価
- **`if let` ガードでコンパイルエラー回避のため遠回りしていた箇所**: 1.95 以降は素直に書けることが多いので、リファクタ候補として記憶しておくと良い

## 今後の注目点

- `cfg_select!` の式コンテキスト利用がエコシステムでどの程度広がるか。`platform-utils` 系クレートのコードベース簡素化に直結する
- 同じく 1.95 で導入された安定 API の活用例（特に `std::sync` 周り）
- `if let` ガードの改善を起点に、パターンマッチ周辺の人間工学的改善が続くか
- `cfg-if` クレート側の今後の方針（標準提供と共存するのか、メンテナンスモードに入るのか）
