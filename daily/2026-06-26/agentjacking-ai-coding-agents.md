# Agentjacking — AI コーディングエージェントを乗っ取る新攻撃

- **日付**: 2026-06-26
- **カテゴリ**: Tech
- **ソース**: [The Hacker News](https://thehackernews.com/2026/06/agentjacking-attack-tricks-ai-coding.html) / [Tenet Security](https://tenetsecurity.ai/blog/agentjacking-coding-agents-with-fake-sentry-errors/)

## 概要

2026年6月、Tenet Security が「Agentjacking」と名付けた新しい攻撃手法を公開した。攻撃者がエラー監視プラットフォーム Sentry に細工したエラーレポートを送り込み、Model Context Protocol（MCP）経由で Sentry を参照する AI コーディングエージェントに、開発者自身の権限で任意のコマンドを実行させるというものだ。

テストでは100以上の組織に対し85%という高い成功率を記録し、少なくとも2,388の組織が悪用可能な DSN（Data Source Name）を露出していた。Claude Code、Cursor、Codex といった主要な AI コーディングアシスタントが、注入されたエラーイベントを MCP 経由で取得し、攻撃者の指示を「正規のデバッグ手順」と解釈して実行してしまうことが確認された。

## 詳細

### 攻撃の仕組み

Sentry はエラーイベントを受け付けるための DSN を持つが、これが公開されているケースが多い。攻撃者はこの公開 DSN を使い、`message` フィールドや context フィールドに悪意ある指示を Markdown として埋め込んだエラーイベントを送り込む。

開発者が AI エージェントに「このエラーを調査して」と依頼すると、エージェントは MCP 経由で Sentry からエラー内容を取得する。このとき、注入された Markdown がエージェントに対する「指示」として解釈され、攻撃者が用意したサーバからコードを取得・実行する、といった一連の操作が自律的に行われてしまう。プロンプトインジェクションが「外部データソース経由」で成立する典型例だ。

### 本質的な問題 — 信頼境界の崩壊

この攻撃が示すのは、MCP の「暗黙の信頼」と Sentry のような「オープンな取り込みアーキテクチャ」の組み合わせが持つ構造的な危うさだ。エージェントは外部から取得したデータ（＝信頼できない入力）と、ユーザーからの正規の指示を区別せずにコンテキストへ取り込んでしまう。データとコードの境界が曖昧になる、という古典的な脆弱性パターンが AI エージェントの文脈で再来している。

### ベンダー対応

Sentry は根本的な修正は「プラットフォームレベルでは技術的に防御不可能」として、抜本対応を見送った。実施されたのは、研究で特定された具体的なペイロード文字列をブロックするコンテンツフィルタのみで、注入を可能にするアーキテクチャ上の経路そのものは塞がれていない。つまり対症療法にとどまっている。

## 今後の注目点

- MCP 仕様側で「信頼できないデータ」と「指示」を分離する仕組み（データの来歴タグ付け、ツール出力のサンドボックス化）が標準化されるか
- AI コーディングエージェント各社（Anthropic / Cursor / OpenAI）が、外部 MCP データの取り扱いにガードレールを追加するか
- Sentry に限らず、外部から書き込み可能なあらゆるデータソース（イシュートラッカー、ログ、コメント欄）が同種の攻撃面になり得る点への注意喚起
- 開発者側の防御策 — エージェントのコマンド実行に承認ゲートを設ける、最小権限で動かす、信頼できる MCP サーバのみ接続する、といった運用の定着
