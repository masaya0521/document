# マネーフォワード GitHub 不正アクセスとサプライチェーン教訓

## 概要

マネーフォワードは 5/1、社員アカウントの GitHub 認証情報が漏えいし、これを悪用した不正アクセスによりプライベートリポジトリがコピーされたと公表した。流出した可能性があるのは子会社が手掛ける法人向け「マネーフォワード ビジネスカード」保有者 370 件の氏名とカード番号下 4 桁、および各種サービスのソースコード一部。CVV・口座連携情報など金融取引に直結する機密データは含まれていないと説明している。直後に「マネーフォワード ME」「マネーフォワード クラウド」では銀行口座連携機能を一時停止し、認証情報の無効化と再発行を順次完了。5/3・5/7 に追加情報を公表した。

国内 PFM・クラウド会計の代表的サービスを抱える事業者が、開発インフラ起点で個人情報・ソースコード両方を露出させた事例として、業界横断のサプライチェーンセキュリティの議論を再燃させている。

## 詳細

### 攻撃経路と被害範囲
- 漏えい起点: 社員に紐づく GitHub 認証情報。パスワード・PAT・SSH キーいずれかとみられる。
- 被害範囲:
  - ビジネスカード保有者 370 件の氏名・カード番号下 4 桁
  - グループサービスのソースコード一部
  - リポジトリに同梱されていたファイル内の個人情報の一部
- 含まれない情報: CVV、有効期限、カード番号全桁、銀行・証券等の連携用ログイン情報。

### 即応した対応
- 認証情報の無効化と再発行を概ね完了。
- 「マネーフォワード ME」「クラウド」両系統で銀行口座連携機能を一時停止し、5/3 13:00 に状況更新。
- 5/7 に被害件数の確定と再発防止策を追記。グループ内子会社（マネーフォワードケッサイ）も別途お知らせを公表。
- ソースコード流出に伴うクレデンシャル類の総点検と、リポジトリ内ハードコードシークレットの掃き出し（secret scanning）を実施中とみられる。

### サプライチェーンとしての論点
1. **GitHub を「開発の生命線」とする企業のリスク表面**: SaaS のクラウド会計・銀行連携事業者は IaC・連携用キー・暗号鍵管理スクリプトなどがコードに集約されやすく、リポジトリ流出はそのまま下流の銀行連携 API キーや内部 IAM の調査対象を増やす。
2. **MFA + ハードウェアキー、SSO 強制、PAT の有効期限短縮**: 一般的ベストプラクティスだが、開発生産性とのトレードオフで PAT を長命にしている組織は多い。今回の事例は短命化と GitHub Apps への置き換えを促す。
3. **secret scanning と push protection の徹底**: GitHub の標準機能である push protection を無効化していたか、過去のコミットに残ったシークレットが既に外部にコピーされた可能性がある。事故後はリポジトリ全履歴のスキャンと回転（rotation）が不可欠。
4. **インシデントレスポンスの透明性**: 5/1, 5/3, 5/7 と段階的に更新しており、国内 PFM 業界としては相対的に開示の頻度・粒度ともに高い水準。米国のディスクロージャー慣行に近づいている。

### 国内コンテキスト
- 2026 年版 IPA「情報セキュリティ 10 大脅威」で「サプライチェーン攻撃」「内部不正」「クラウドサービスの設定ミス」がいずれも上位。GitHub 経由の認証情報漏えいは「サプライチェーン経由の不正アクセス」と「開発環境の権限管理」に該当。
- 金融機関・カード会社の API 連携は、金融庁の監督下にあり、銀行口座連携の停止は短期的に PFM 利用者の家計簿同期に支障をきたす。
- 同社は IPO 後の事業多角化（クラウド会計・電帳法対応・経理 BPO）でリポジトリ数が急増しており、シークレット管理の運用設計が試される段階。

## 今後の注目点

- 「ソースコード一部流出」の具体的範囲が後日追加開示されるか（特に銀行連携モジュールの含有有無）。
- 流出経路の詳細：被害アカウントの認証要素（PAT/Web パスワード/SSO セッション）と、フィッシング・端末感染・PAT 漏れのいずれだったか。
- 銀行口座連携機能の再開タイミングと、再発防止策の具体（ハードウェアキー必須化、GitHub Apps 化、push protection 強制、機密リポジトリの IP 制限など）。
- 国内クラウド会計・PFM 各社の追随対応（freee、弥生、SBI 関連サービス等）と業界自主基準の動き。
- 個人情報保護委員会への報告・公表内容、行政指導・是正命令の有無。
- 2026 年版 IPA 10 大脅威の改訂時に本件がケーススタディとして取り上げられるか。
