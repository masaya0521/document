# マネーフォワード GitHub 不正アクセス事件 — 認証情報漏えいとサプライチェーン攻撃の構造

- **日付**: 2026-05-11
- **カテゴリ**: Japan Tech
- **ソース**: [日経新聞 - マネーフォワード、不正アクセスで個人情報370件流出](https://www.nikkei.com/article/DGXZQOUB012UA0R00C26A5000000/) / [マネーフォワード ME 公式お知らせ](https://support.me.moneyforward.com/hc/ja/articles/57504390625305) / [マネーフォワード クラウド 公式お知らせ](https://biz.moneyforward.com/support/news/20260501.html)

## 概要

マネーフォワードは 2026 年 5 月 1 日、ソフトウェア開発・システム管理に利用している GitHub の認証情報が漏えいし、第三者によってリポジトリがコピーされたと発表した。流出が確認されたのは子会社の「マネーフォワードビジネスカード」保有者 370 件分の氏名とカード番号下 4 桁。事案発覚後、各提携金融機関との安全性確認のため、銀行口座連携機能を一時停止する措置を取り、5/3・5/7 と段階的に続報を出している。フィンテック中核企業の GitHub アカウント侵害という構図は、同時期に世界で頻発する SaaS・OAuth 経由のサプライチェーン攻撃と並ぶ事案として注目される。

## 詳細

### 攻撃の入口と被害範囲

公開情報を整理すると、攻撃の起点は GitHub の「認証情報の漏えい」であり、SSH 鍵・Personal Access Token・OAuth トークンのいずれか（あるいは複数）が第三者の手に渡ったと推定される。攻撃者はその資格情報を使って同社のリポジトリをクローンしている。

確認されている流出データは以下の通り。

- マネーフォワードビジネスカード保有者 370 件の「氏名」+「カード番号下 4 桁」
- カード番号の全桁、有効期限、CVV は流出していないと公表
- ソースコードについては「流出した可能性」と表現され、確定的な記述は避けられている

### 二次被害のリスク

ソースコード流出が現実化した場合の二次リスクは大きく、以下が想定される。

- **未公開脆弱性の発見**: 内部実装の解析により、外部からは見えないバリデーション欠落・ロジック欠陥が特定される
- **認証フロー迂回**: OAuth・SSO・MFA 実装の弱点を理解した上での認証バイパスの試行
- **API の悪用**: 内部 API のレートリミット・スコープ設計の解析を踏まえた濫用
- **秘密情報のハードコード露出**: 仮にリポジトリ内に環境変数・接続文字列・API キーが残っていた場合、その悪用

マネーフォワードはフィンテック中核企業として、銀行・カード会社・会計事務所への API 接続を多数保有しており、ソースコードに含まれる「連携先側の仕様情報」が二次的な攻撃面を広げる可能性も無視できない。

### 世界的な GitHub 経由のサプライチェーン攻撃の流れの中での位置付け

2025-2026 年は GitHub の OAuth Apps・Personal Access Token・自動化トークンを起点とした侵害が世界的に増加している。Cushman & Wakefield の Salesforce 経由 50 万件流出（同 5 月）など、SaaS の信頼チェーンを辿る攻撃が常態化しており、本件もその文脈に位置付けられる。

### 同社の対応プロセスから読み取れるシグナル

- **連携停止という劇薬の早期投入**: 銀行口座連携を「全停止」した判断は、ユーザビリティの大幅な毀損と引き換えに「安全側」に振り切った決断。フィンテック企業として正しい優先順位を示した
- **段階的な情報公開（5/1 → 5/3 → 5/7）**: 確定情報のみを順次出すスタイル。早期に断定的な発表をして後で訂正するパターンを避けている
- **再発行と確認作業の完了を再開条件として明示**: 「認証情報の再発行などすべての確認作業が完了した後にサービスを順次再開」という表現で、ユーザーへの説明責任を保ちつつ復旧時期は曖昧に保留

## 今後の注目点

- **流出経路の最終公表**: 「GitHub の認証情報がどのように漏えいしたか」（フィッシング・端末侵害・サードパーティ OAuth App など）が後日公開されれば、業界全体の対策指針が明確化する
- **ソースコード流出の確定情報**: 「可能性」表現が「確定」「不発」のいずれに収束するかで、二次攻撃のフィージビリティが決まる
- **銀行口座連携機能の再開タイミング**: 連携先金融機関の確認プロセスの長さで「フィンテック × 安全確認」のリードタイムが可視化される
- **金融庁・個人情報保護委員会の対応**: 個人情報保護法の重大事案として、報告内容や是正命令の有無
- **国内フィンテック企業の防御強化**: 同業他社（freee、Stripe、Visa Japan、PayPay 等）の GitHub・OAuth Apps 周辺の対策の見直し動向
- **OAuth Apps Workflow への影響**: GitHub 側が認証 App の権限分離・監査ログ強化を急ぐきっかけになるか
