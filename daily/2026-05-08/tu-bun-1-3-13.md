# Bun v1.3.13 — テストランナーとインストーラの大幅改善

- **日付**: 2026-04-19 リリース（記録: 2026-05-08）
- **カテゴリ**: Backend / Build Tool（JavaScript ランタイム）
- **ソース**: [Bun Blog v1.3.13](https://bun.com/blog/bun-v1.3.13), [GitHub Releases](https://github.com/oven-sh/bun/releases)

## 概要

Bun v1.3.13 は 82 件の issue を解決した中規模アップデートで、特に **テストランナーの並列実行** と **インストーラのメモリ効率改善** が目玉となる。`bun test` には `--parallel` / `--isolate` / `--shard` / `--changed` の 4 つのフラグが新たに加わり、大規模プロジェクトでのテスト実行体験が大きく変わる。`bun install` 側もダウンロード中のストリーム展開へ切り替わり、メモリ使用量を従来の 1/17 まで削減した。

その他、ソースマップのメモリ消費が 1/8 に圧縮され、`zlib-ng` ベースの gzip が 5.5 倍高速化、`Bun.serve()` が Range リクエストへ対応するなど、地味ながら実運用で効くチューニングが詰まっている。

## 主な変更点

### `bun test` の並列・分割実行

- `--parallel`: テストファイルを最大 N プロセスに分散実行（デフォルトは CPU 数）。Linux では cgroup-aware で適切な並列数を選択する。
- `--isolate`: 各テストファイルを同一プロセス内で fresh な global 環境で実行。グローバル汚染による flaky test を抑制しつつ、プロセス起動コストを払わない中間策。
- `--shard`: CI を複数ジョブに分散する際の shard 指定（`1/4` など）。
- `--changed`: git の差分から影響を受けるテストだけを実行。ローカル開発のフィードバックループを縮める。

```bash
# CPU 数に応じてテストを並列化
bun test --parallel

# CI で shard 1/4 を実行
bun test --shard=1/4

# ローカルで変更されたファイルに関係するテストのみ
bun test --changed
```

### `bun install` のストリーム展開

従来は `.tgz` をメモリにバッファし、解凍した `.tar` もメモリに置いてから展開していた。v1.3.13 ではダウンロード中のストリームから直接ディスクへ展開する方式へ切り替わり、**メモリ使用量を 17 倍削減**。大量のパッケージを一括インストールするモノレポや CI 環境で効果が大きい。

### パフォーマンス改善

- ソースマップのメモリ消費が 1/8 に。
- gzip が `zlib-ng` 採用で 5.5 倍高速化。
- WebKit JavaScriptCore の上流から 1,316 commit を取り込み（v1.3.12 時点）。

### ランタイム機能追加

- `Bun.serve()` が HTTP Range リクエストをサポート（動画ストリーミングや部分ダウンロードで重要）。
- `node:crypto` および WebCrypto に SHA3 が追加。
- `ws+unix://` スキームで UNIX ドメインソケット越しの WebSocket クライアントが利用可能に。
- 多数の Node.js 互換性向上。

## 破壊的変更・移行ガイド

破壊的変更は明示されていない。`--parallel` / `--isolate` はオプトインのフラグで、既存の `bun test` コマンドの挙動は変わらない。CI で並列を有効にする場合は、テストが本当に独立しているかを確認すること（共有リソース・ファイルシステム・ポート衝突など）。

## 今後の注目点

- `--isolate` と `--parallel` の組み合わせがどこまで Jest/Vitest の置き換え候補になるか。Node.js テストランナーや Vitest は引き続き並列性で先行しているが、Bun のシングルバイナリ・速度の優位は CI で訴求力が大きい。
- `bun install` のストリーム展開が pnpm/npm/yarn に対してメモリ・速度面でどこまでリードを広げるか。
- Bun v1.4 系（ロードマップ未公開）でルーターや DB クライアント周りがどう発展するか。InfoQ では v3.1 系で「組み込み DB クライアント」「zero-config フロントエンド開発」が取り上げられているが、現行 1.3 系の延長線上にどう統合されるかは要観測。
