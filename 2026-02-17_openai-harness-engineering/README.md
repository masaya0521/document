# Harness engineering: leveraging Codex in an agent-first world

- **出典**: https://openai.com/index/harness-engineering/
- **発行元**: OpenAI
- **確認日**: 2026-02-17

## 概要

OpenAI の内部チームが **5ヶ月間、人間が手動でコードを一切書かず**、Codex エージェントだけで新製品を構築した経験から得られた知見をまとめた記事。約 **1,500 のプルリクエスト** を通じて **約100万行のコード** が生成され、手書きと比較して **約10倍の開発速度** を実現したと報告されている。

## 主な知見

### 1. エンジニアの役割の変化

- エンジニアの主な仕事は「コードを書くこと」から「エージェントが有効に作業できる環境を整えること」に変わった
- システム設計、スキャフォールディング（足場作り）、レバレッジの確保が中心的な仕事になる

### 2. 初期の進捗は予想より遅かった

- Codex の能力不足ではなく、**環境が未整備だったこと**が原因
- エージェントが高レベルな目標に向けて進むために必要なツール・抽象化・内部構造が不足していた

### 3. コンテキスト管理 — 「地図を渡せ、1,000ページのマニュアルではなく」

- コンテキストは希少資源であり、巨大な指示ファイルはタスクやコード、関連ドキュメントを圧迫する
- `AGENTS.md` は百科事典ではなく **目次（テーブルオブコンテンツ）** として扱うべき（約100行程度）
- リポジトリの知識ベースは構造化された `docs/` ディレクトリに配置し、`AGENTS.md` はそこへのポインタとして機能させる

### 4. アーキテクチャの厳格な制約

- 各ビジネスドメインを固定のレイヤーに分割し、依存関係の方向を厳密に検証
- 通常は数百人規模のチームで初めて導入されるような制約だが、エージェント開発では **初期段階から必要**
- これにより、アーキテクチャのドリフト（逸脱）なく高速開発が可能になる

### 5. カスタムリンターと CI による強制

- 構造化ログ、スキーマ・型の命名規則、ファイルサイズ制限、プラットフォーム固有の信頼性要件などをカスタムリンターと構造テストで強制
- エラーメッセージには **修正手順をエージェントのコンテキストに注入する** 仕組みを採用

## 核心的なメッセージ

ソフトウェア構築にはなお規律が求められるが、その規律が発揮される場所は **コードそのものではなくスキャフォールディング（足場）側** に移行している。ツール、抽象化、フィードバックループの重要性がますます高まっている。
