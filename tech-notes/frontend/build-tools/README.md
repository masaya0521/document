# フロントエンドビルドツールの全体像

JavaScript のモジュールシステムの歴史から、現代のビルドツール（Vite, Rollup, esbuild, Rolldown）までを整理する。

## 1. JavaScript モジュールシステムの歴史

初期の JavaScript にはモジュールの仕組みが存在せず、すべてがグローバルスコープに置かれていた。

```html
<!-- 昔のやり方：スクリプトの読み込み順序に依存 -->
<script src="jquery.js"></script>
<script src="utils.js"></script>
<script src="app.js"></script>
```

この方式には名前衝突、依存関係の不明確さ、読み込み順序への依存という問題があった。

### モジュールシステムの変遷

| 時期 | 仕様 | 環境 |
|---|---|---|
| 2009年 | **CommonJS** (`require`/`module.exports`) | Node.js |
| 2011年 | **AMD** (`define`/`require`) | ブラウザ（RequireJS） |
| **2015年** | **ES Modules** (`import`/`export`) | **ECMAScript 言語仕様として標準化** |

CommonJS や AMD はコミュニティが作った非公式な仕様。ES Modules は ECMAScript に正式に組み込まれた公式のモジュールシステム。

## 2. ES Modules（ESM）

```js
// math.js — エクスポート
export function add(a, b) {
  return a + b
}

// app.js — インポート
import { add } from './math.js'
console.log(add(1, 2)) // 3
```

### ESM の特徴

- **静的解析が可能** — `import`/`export` はファイルの先頭に書く必要があり、実行前に依存関係が確定する。これにより Tree Shaking（未使用コードの除去）が可能
- **非同期読み込み** — ブラウザがモジュールを非同期で取得できる
- **スコープが独立** — 各モジュールは独自のスコープを持ち、明示的に export したものだけが外部から参照可能

## 3. CommonJS の制約

Node.js のサーバーサイド用途としてはうまく機能したが、以下の制約がある。

### 動的な require

```js
const mod = require(condition ? './foo' : './bar')
const lib = require(`./plugins/${name}`)
```

`require` は普通の関数なので変数や条件分岐の中で自由に使える。柔軟だが、実行するまでどのモジュールを読み込むか分からないため：

- **Tree Shaking ができない** — 使われていないコードを事前に判別できず、バンドルサイズが大きくなる
- **静的解析ツールが困る** — エディタの補完やリンターが依存関係を正確に把握できない

### 同期読み込み

```js
const data = require('./heavy-module') // ここで処理がブロックされる
```

Node.js ではファイルシステムから読むだけなので問題になりにくいが、ブラウザではネットワーク越しに取得する必要がある。同期読み込みはブラウザの描画をブロックしてしまうため、CommonJS はブラウザで直接動かせない。

### exports の挙動が分かりにくい

```js
// これは動く
exports.foo = 'bar'

// これは動かない（参照が切れる）
exports = { foo: 'bar' }

// こう書く必要がある
module.exports = { foo: 'bar' }
```

`exports` は `module.exports` への参照に過ぎないため、再代入すると壊れる。

## 4. バンドラーの系譜

### Webpack（2012年〜）

CommonJS 時代に生まれたバンドラー。各モジュールを関数でラップし、独自のランタイムで結合する方式。

- 開発サーバー起動時に全モジュールの依存関係を解析し、全体をバンドルしてからブラウザに配信
- CSS・画像など非 JS アセットの処理も標準で豊富
- プロジェクトが大きくなるほど起動・ビルドが遅くなる

### Rollup（2015年〜）

Rich Harris（Svelte の作者）が開発。ES Modules を前提に設計されたバンドラー。

```js
// src/utils.js
export function add(a, b) { return a + b }
export function sub(a, b) { return a - b }  // 未使用

// src/app.js
import { add } from './utils.js'
console.log(add(1, 2))
```

Rollup の出力（フラットに結合、未使用の sub は消える）：

```js
function add(a, b) { return a + b }
console.log(add(1, 2))
```

Webpack の出力（モジュールごとに関数でラップされ、ランタイムが付く）：

```js
(function(modules) {
  // webpack ランタイム...
})({
  './utils.js': function(exports) { ... },
  './app.js': function(exports) { ... }
})
```

| | Rollup | Webpack |
|---|---|---|
| Tree Shaking | 優秀（設計の根幹） | 後から対応、Rollup ほど徹底されていない |
| 出力サイズ | 小さい | ランタイム分だけ大きい |
| ライブラリのビルド | 得意 | やや冗長 |
| 開発サーバー・HMR | 自前では持たない | 充実 |

React や Vue など多くの OSS ライブラリが Rollup でビルドされている。

### esbuild（2020年〜）

Evan Wallace（Figma の共同創業者）が開発。Go 製のバンドラー・トランスパイラ。

- JavaScript/TypeScript のバンドル・トランスパイル・ミニファイを高速に行う
- **型チェックをしない**。TypeScript の構文を理解して除去するだけ

```ts
// esbuild はこの型注釈を単に消すだけ
const x: number = 1  →  const x = 1
```

### Rolldown（Vite 8 で統合）

Vite チーム（Evan You が主導）が開発。Rust 製の Rollup 後継。Vite 8.0（2026年3月リリース）で正式統合された。

- Rollup 互換 API でプラグインエコシステムをそのまま活かせる
- トランスパイルも内蔵（内部で Oxc パーサーを使用）し、esbuild の役割も兼ねる
- Rollup 比で 10-30x の高速化を実現

## 5. Vite

Evan You（Vue.js の作者）が開発したフロントエンド開発用のビルドツール。

### アーキテクチャ

**開発時（ネイティブ ESM 方式）：**

1. 開発サーバー起動時にはバンドルしない
2. ブラウザが `<script type="module">` で直接 `import` 文を解釈
3. ブラウザがモジュールを要求した時点で、そのファイルだけを変換・配信（esbuild で変換）

→ 起動時のコストがほぼゼロ。プロジェクト規模に依存しにくい高速 HMR。

**本番ビルド：**

Rollup でバンドル。ネイティブ ESM をそのまま本番で使うと HTTP リクエスト数が膨大になりパフォーマンスが悪化するため。

### Webpack との比較

| | Webpack | Vite |
|---|---|---|
| 開発サーバー起動 | 遅い（全体バンドル） | 速い（オンデマンド） |
| HMR | プロジェクト規模に比例して遅くなる | 規模に依存しにくい |
| 設定 | 複雑になりがち | シンプル |

### バージョンごとのバンドラー構成

```
Vite 2〜7:  esbuild（開発時変換） + Rollup（本番バンドル）
Vite 8:     Rolldown（両方を統一）
```

Vite 8 で Rolldown に統一されたことで、開発と本番の挙動差異が解消され、ビルド速度も大幅に向上した。

## 6. tsgo — TypeScript コンパイラの Go 実装

Microsoft の TypeScript チーム公式による、`tsc` の Go ネイティブ実装。大規模プロジェクトでの型チェックを10倍程度高速化することを目指している。

### esbuild との違い

| | esbuild | tsgo |
|---|---|---|
| 言語 | Go | Go |
| 役割 | バンドラー・トランスパイラ | 型チェッカー |
| TypeScript の型を検査するか | **しない** | **する（本業）** |
| 出力 | バンドルされた JS ファイル | 型エラーの報告、`.d.ts` 生成 |
| 置き換え対象 | Webpack, Rollup など | `tsc` |

esbuild が速いのは「型チェックを捨てた」から。tsgo は「型チェックそのものを速くする」アプローチ。両者は補完関係にあり、競合するものではない。

```
エディタ上の型チェック  →  tsc（将来は tsgo）
ビルド・バンドル        →  esbuild / Vite / Rollup
```

## 7. 各ツールの設計思想と言語選択

### esbuild がなぜ速いか

esbuild の速さは Go 言語の特性と設計判断の両方から来ている。

**言語レベルの優位性：**

- Go はコンパイル言語でネイティブバイナリに変換される。JavaScript（Rollup）のようにランタイムの解釈コストがない
- Go のゴルーチンによる並列処理が自然に書ける。ファイルのパース・変換を複数コアで同時実行できる

**設計判断：**

- **シングルパス設計** — ソースコードを1回走査するだけでパース・変換・出力を行う。多段パイプラインを避けている
- **AST を最小限しか作らない** — 完全な AST（抽象構文木）を構築せず、必要な情報だけを保持して変換する
- **プラグインの柔軟性を犠牲にしている** — Rollup のような自由度の高いプラグインフックを提供しない代わりに、内部最適化を優先

つまり「**速度のために柔軟性を割り切った**」設計。

### Rollup がなぜ最適化に優れるか

**Tree-shaking の精度：**

- Rollup は ES Modules の `import`/`export` を静的に解析する前提で設計された最初期のバンドラー
- ES Modules は静的構造（`import` は必ずトップレベル、動的に変化しない）なので、「この export は誰にも使われていない」を正確に判定できる
- **文レベル**ではなく**バインディング（変数・関数）レベル**で不要コードを除去できる。webpack が当初ファイル単位でしか除去できなかったのと対照的

**プラグインアーキテクチャ：**

- ビルドパイプラインの各段階（解決 → 読み込み → 変換 → 出力）に細かいフックを提供
- プラグインが AST を自由に操作できるため、高度な最適化やカスタム変換が可能
- この柔軟性がエコシステムの豊富さにつながった

**代償：**

- JavaScript で書かれているので、パース・AST 操作・コード生成すべてが JS のシングルスレッドで動く
- プラグインフックを順番に呼び出すオーバーヘッドも積み重なる

つまり「**柔軟性と最適化品質のために速度を犠牲にした**」設計。

### Rolldown がなぜ両立できるか

Rolldown は「esbuild と同じことを Rollup の設計で」ではなく、**両者の制約を別の手段で解決**している。

**esbuild の速度を得る手段：**

- **Rust 製** — Go と同様にネイティブバイナリにコンパイルされる。さらに Rust はゼロコスト抽象化とメモリ安全性を両立しており、GC の停止がない
- **Oxc パーサー** — Rust 製の高速 JavaScript/TypeScript パーサーを内部で使用。パース自体がネイティブ速度
- **並列処理** — Rust の `rayon` 等によるデータ並列処理で、複数ファイルを同時にパース・変換

**Rollup の最適化品質を得る手段：**

- **Rollup 互換の Plugin API** — Rollup と同じフック体系（`resolveId`, `load`, `transform`, `renderChunk` 等）を実装。既存プラグインがそのまま動く
- **ES Modules ベースの Tree-shaking** — Rollup と同じ静的解析アプローチを Rust で再実装。精度を維持しつつネイティブ速度で実行
- **完全な AST 構築** — esbuild が省略していた完全な AST を構築する。ただし Rust のメモリ効率により、JS で AST を構築するよりはるかに軽い

| 制約 | esbuild の解法 | Rollup の解法 | Rolldown の解法 |
|------|---------------|--------------|----------------|
| 速度 | Go + 簡略化 | — (遅い) | Rust + Oxc + 並列処理 |
| Tree-shaking 精度 | — (簡易的) | 完全な AST 静的解析 | Rust で同等の解析を再実装 |
| プラグイン柔軟性 | 最小限 | 豊富なフック体系 | Rollup 互換フックを Rust で実装 |
| AST 操作コスト | AST を省略 | JS での AST 操作は重い | Rust の AST はメモリ効率が高い |

esbuild が「速度のために割り切った」部分と、Rollup が「JS だから遅かった」部分を、**Rust という言語の特性で両方解決した**のが本質。「両者のいいとこ取り」というより、「両者が抱えていた制約がどちらも言語起因だったので、適切な言語で再実装したら両立できた」。

### tsgo が Go、Rolldown が Rust を選んだ理由

同じ「JavaScript ツールチェインのネイティブ化」でも、言語選択が分かれた理由はデータ構造の性質の違いにある。

**tsgo（型チェッカー）が GC のある Go を選んだ理由：**

- 型チェックでは AST ノード同士が複雑に**相互参照**する（型Aが型Bを参照し、型Bが型Cを参照し、型Cが型Aを参照…）
- 型推論の過程で**大量の中間オブジェクト**が生成され、どれがいつ不要になるか事前に判定しづらい
- オブジェクトの**ライフタイムが不規則** — あるノードが突然別の場所から参照されることがある
- Rust の所有権モデルでこれを表現すると `Rc<RefCell<T>>` やアリーナアロケータを多用することになり、コードが複雑化してパフォーマンスも出にくい

**Rolldown（バンドラー）が所有権モデルの Rust で済む理由：**

- バンドラーの処理は**パイプライン型** — パース → 変換 → Tree-shaking → コード生成、と段階が明確
- 各ファイルの AST は基本的に**独立**しており、ファイル間の相互参照は `import`/`export` の解決時だけ
- 処理段階ごとに「このデータはもう不要」が**明確に決まる** — パースが終われば生テキストは不要、変換が終われば元の AST は不要
- ファイル単位の並列処理が自然にできる（ファイル間でデータを共有する場面が少ない）

| | 型チェッカー (tsgo) | バンドラー (Rolldown) |
|---|---|---|
| データ構造 | 複雑な相互参照グラフ | ファイル単位の独立した AST |
| ライフタイム | 不規則、予測困難 | パイプライン段階で明確 |
| 並列化の単位 | 難しい（型の依存が絡む） | ファイル単位で自然 |
| 適した言語 | GC あり（Go） | 所有権モデル（Rust） |

「GC vs 所有権」は単純にどちらが優れているではなく、**扱うデータの参照パターンとライフタイムの予測しやすさ**で適性が分かれる。
