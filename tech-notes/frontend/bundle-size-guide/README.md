# フロントエンドのバンドルサイズ規模感ガイド

## この文書が答える問い

「JavaScript のバンドルサイズにおいて、ある追加が "大きい" か "小さい" かを判断する基準は何か」

## 結論

- 一般的な Web アプリの JS バンドルは gzipped で 200-400KB 程度
- React 本体が ~45KB gzipped なので、それと同程度のライブラリは「かなりの割合を占める」と言える
- ライブラリの追加判断は、機能の重要度とサイズのバランスで行う

---

## 1. 主要ライブラリのサイズ一覧

| ライブラリ | gzipped | 用途 |
|-----------|---------|------|
| React + ReactDOM | ~45KB | UI フレームワーク本体 |
| Vue.js | ~33KB | UI フレームワーク本体 |
| lodash (全体) | ~25KB | ユーティリティ |
| date-fns (全体) | ~20KB | 日付操作 |
| zod | ~14KB | バリデーション |
| axios | ~5KB | HTTP クライアント |
| OTel SDK (web) | ~30-50KB | トレーシング |
| Zone.js | ~13KB | 非同期コンテキスト追跡 |

## 2. アプリ全体の目安

| 規模 | gzipped サイズ | 備考 |
|------|---------------|------|
| 軽量 | ~100KB | 目標として理想的 |
| 標準的 | 200-400KB | 多くのプロダクション SPA |
| 重め | 500KB+ | 改善の余地あり |

## 3. サイズが体験に与える影響

バンドルサイズが大きいと:

- **初回ロード時間の増加**: ダウンロード + パース + 実行のすべてに影響
- **モバイル環境への影響が大きい**: 低速回線・低スペック端末では顕著
- **キャッシュ効率の低下**: バンドルが変わるたびに再ダウンロードが発生

gzipped 100KB の JS は、3G 回線で約 3 秒、4G 回線で約 0.5 秒のダウンロード時間に相当する。

## 4. 判断の考え方

ライブラリを追加する際の判断基準:

- **~5KB 以下**: ほぼ気にしなくてよい
- **5-20KB**: 用途に見合えば許容範囲
- **20-50KB**: コア機能でなければ代替手段を検討する価値がある
- **50KB+**: 明確な理由がなければ避けたい。サーバーサイドに寄せられないか検討

補助的な機能（ロギング、トレーシング、アナリティクス等）にコア機能（React 等）と同程度のサイズを使うのは、費用対効果の観点から慎重に判断すべき。SSR/BFF 構成でサーバーサイドに寄せられる処理は、バンドルに含めない選択肢を優先する。
