# メモリアロケータ、GC、所有権モデルの比較

- **日付**: 2026-03-27
- **カテゴリ**: Language / Memory Management / Performance

## 1. 概要

メモリ管理戦略には大きく「手動管理 (C)」「所有権モデル (Rust)」「GC (Go, Java, JS)」の 3 つがある。本ドキュメントでは、それぞれのフラグメンテーション特性、パフォーマンス影響、および mimalloc のようなカスタムアロケータの役割を整理する。

---

## 2. メモリフラグメンテーション

### フラグメンテーションとは

確保と解放を繰り返すうちに、空きメモリが細切れになり、合計では十分な空きがあるにもかかわらず大きな連続領域を確保できなくなる現象。

```
メモリ: [A][B][C][D][E]
  ↓ B と D を解放
メモリ: [A][空][C][空][E]
  ↓ B+D 分の空きはあるが、連続した大きなブロックは確保できない
```

### GC の有無との関係

フラグメンテーションの発生原因は「確保と解放のタイミング・サイズがバラバラ」であること自体にあり、GC の有無とは直接関係しない。むしろ、手動管理 (Rust/C) の方がフラグメンテーションに対して不利な面がある。

- **GC 言語 (Java, Go 等)**: コンパクション GC はオブジェクトをメモリ上で移動し、断片化を解消できる
- **Rust/C**: ポインタが直接アドレスを参照しているため、オブジェクトを勝手に移動できない。フラグメンテーションが起きても解消手段がない

```
GC によるコンパクション:
[A][空][C][空][E]  →  [A][C][E][空空空]
                        ↑ 連続した空き領域ができる
```

## 3. mimalloc: カスタムアロケータによるフラグメンテーション軽減

[mimalloc](https://github.com/microsoft/mimalloc) は Microsoft が開発した汎用メモリアロケータ。コンパクションができない手動管理言語で、アロケータ側の工夫によりフラグメンテーションを軽減する。

### 主な設計

- **サイズクラス別のページ管理**: 同じサイズの割り当てを同じページにまとめ、解放後の穴を再利用しやすくする
- **スレッドローカルなフリーリスト**: スレッド間でメモリプールを分離し、ロック競合と断片化を同時に軽減

### Rust での使い方

`#[global_allocator]` アトリビュートでデフォルトアロケータを差し替える。アプリケーションコードの変更は不要。

```rust
#[global_allocator]
static GLOBAL: mimalloc::MiMalloc = mimalloc::MiMalloc;
```

## 4. GC のデメリット

フラグメンテーション解消という強みがある一方、GC には以下のデメリットがある。

### 4.1 Stop-the-World による一時停止

コンパクションやマーク処理のためにアプリケーションスレッドを一時停止する必要がある。Go や ZGC (Java) など最近の GC は停止時間を極限まで短くしているが、ゼロにはできない。

- Web サーバーで数十ms〜数百ms のレスポンス遅延が突然発生する
- リアルタイム処理 (ゲーム、音声等) ではフレーム落ちや音飛びの原因になる

### 4.2 メモリのオーバーヘッド

GC が効率的に動くためには、実際に必要な量よりかなり多くのメモリが必要。

- オブジェクトごとのヘッダ (型情報、GC マーク用ビット等)
- コンパクションには移動先の空き領域が必要
- GC の実行頻度を下げるにはヒープに余裕を持たせる必要がある

一般に GC 言語は同等の処理を行う Rust/C と比べて 2〜5 倍のメモリを消費する。

### 4.3 解放タイミングが非決定的

```rust
// Rust: スコープを抜けた瞬間に解放
{
    let file = File::open("data.txt")?;
}  // ← ここで確実に close

// Java: いつ解放されるか不定
{
    var file = new FileInputStream("data.txt");
}  // ← GC が回収するまで file は開いたまま
```

ファイルハンドル、DB コネクション、ロックなど「メモリ以外のリソース」は早く解放したいが、GC 任せだとタイミングが制御できない。Java の `try-with-resources` や Python の `with` はこの問題への対処。

### 4.4 スループットへの影響

GC 自体が CPU 時間を消費する。並行 GC であってもバックグラウンドで CPU を使い、アプリケーション本来の処理に使えるリソースが減る。

## 5. GC が有利なケース: コンパイラのワークロード

GC のデメリットが問題にならず、むしろ利点が活きるケースもある。代表例がコンパイラ。

### TypeScript の Go 書き直し (tsgo)

TypeScript チームは TypeScript コンパイラを Go で書き直し、10 倍のパフォーマンス改善を達成した。Rust ではなく Go を選んだ理由:

- **割り当てパターンとの相性**: コンパイラは AST ノード、型情報、スコープ情報など大量の短命オブジェクトをグラフ状に参照し合う。Rust の所有権モデルだと `Rc`/`Arena` を多用することになり、コードが煩雑になる
- **GC の恩恵が大きい**: 大量に作って最後にまとめて捨てるパターンは、GC が最も得意とするワークロード
- **移植のしやすさ**: TypeScript (JS) と Go は GC 言語同士なので、メモリ管理の考え方をそのまま持ち込める

元が JS (インタプリタ) → Go (ネイティブコンパイル + GC) であるため、GC のコストを差し引いても圧倒的に速い。Rust で書けばさらに速くなる可能性はあるが、グラフ構造を扱うコードでの開発コストを考えると Go は現実的に良い選択だった。

## 6. まとめ: メモリ管理戦略の使い分け

| 観点 | Rust (手動/所有権) | GC 言語 (Go, Java) |
|---|---|---|
| フラグメンテーション | アロケータで軽減 (mimalloc 等) | コンパクション GC で解消可能 |
| メモリ効率 | 高い | 2〜5 倍のオーバーヘッド |
| レイテンシの安定性 | GC 停止なし | Stop-the-World のリスク |
| リソース解放 | 決定的 (スコープ終了時) | 非決定的 |
| グラフ構造の扱い | 所有権の制約で煩雑になりやすい | GC が参照管理を担うので自然に書ける |
| 適したワークロード | サーバー、リアルタイム処理 | コンパイラ、短命オブジェクト大量生成 |

**Rust + mimalloc** はサーバーアプリケーションで特に有効。レイテンシの安定性とメモリ効率の両方で GC 言語を上回る。一方、コンパイラのようなグラフ構造を大量に生成・破棄するワークロードでは、GC 言語の方が開発効率・実行効率ともに合理的な選択になりうる。
